M5 · 网络基础与安全

网关 / 端口 / HTTPS / 防火墙 / VPN —— 用生活类比一次性讲清楚

时长
约 20 分钟
关键词
IP、网关、端口、HTTPS、防火墙、VPN、API Key 安全

引言：为什么你需要知道这些？

你不是网络工程师，但你需要知道你的数据怎么流动、有什么风险、怎么保护 API Key。下面四个"层"，从最基本的 IP 到最实用的 VPN，用生活类比一次性讲清楚。

核心类比：你家 公司
你把整个互联网想象成"从你家去公司"的旅程：
IP = 门牌号（你要去哪栋楼）
网关 = 小区大门（你从哪个出口出去）
端口 = 部门分机号（进到楼里找哪个办公室）
HTTPS = 密封信封（你寄的东西别人能不能看）
防火墙 = 门禁（谁可以进来）
VPN = 国际快递中转站（你走哪条路过去）

后面我们逐一展开。

1. 第一层：IP 与网关

时长
约 5 分钟
类比
IP = 门牌号，网关 = 小区大门

IP 地址：互联网上的"门牌号"

互联网上的每一台设备（你的电脑、手机、服务器）都有一个唯一的 IP 地址——就像每个建筑物有一个门牌号。比如 192.168.1.100 或 10.0.0.1。

公网 IP vs 私有 IP：

概念类比科研意义
公网 IP 你家的真实地址（快递员能找到） 你的学校/机构的对外地址；DeepSeek 服务器也有一个公网 IP
私有 IP 同一栋楼内的房间号（只有楼里的人知道） 你实验室电脑的 192.168.x.x——外界看不到，只有校园网内的设备能找到
IPv4 vs IPv6

类比说明
IPv4老式 6 位数座机号全球只有约 43 亿个地址（早已不够用）
IPv616 位手机号地址数量多到可以给地球上每一粒沙分配一个 IP

网关：小区的"大门"

网关 = 小区大门
你家住在 XX 小区 3 栋 501。快递员送快递，先找到小区大门（网关），然后由门卫告诉你"3 栋在那边"。

在互联网中：你的数据包要访问外网（比如 DeepSeek 的服务器），必须先经过你所在网络的"大门"——网关。网关的地址通常是 192.168.1.1 或 10.0.0.1。

在你的科研中：当你调用 DeepSeek API 时，请求的路线是：
你的电脑（192.168.x.x 私有 IP） 学校网关（公网 IP） 互联网骨干 DeepSeek 服务器。

2. 第二层：端口

时长
约 5 分钟
类比
端口 = 公司分机号

端口：你打总机号码之后，要拨的分机号

端口 = 公司分机号
你打"010-12345678"找到了一家公司的大门。但前台接起来会问："您找哪个部门？销售部按 1、技术部按 2、财务部按 3。"

IP 地址让你找到这台服务器。端口号让你找到服务器上运行的具体服务。

同一台服务器同时运行着：网站服务（端口 80）、加密网站（端口 443）、文件传输（端口 22）、数据库（端口 3306）。端口号就是区分"你找的是哪个服务"。

你必须认识的常见端口

端口服务类比科研中的例子
80 HTTP（网页） 公司前台——处理普通访客 访问一个非加密网站（不推荐用于科研数据传输）
443 HTTPS（加密网页） 公司 VIP 通道——加密验证身份 访问 DeepSeek、ChatGPT、任何安全 API
22 SSH（远程登录） 给你一把万能钥匙，从远处打开自己的办公室 远程连接学校服务器跑计算任务
3000 开发预览 内部演示室 WorkBuddy 本地预览 localhost:3000——3000 就是端口号
8080 备用 HTTP 公司侧门 很多科研软件的管理后台使用此端口
端口扫描：坏人用自动拨号器试你的每一个分机号

有一种网络攻击叫"端口扫描"：攻击者对你的 IP 地址，从端口 1 到端口 65535 逐个测试——看看哪个端口开着、哪个端口跑的是什么服务、哪个服务有已知漏洞。

好消息是：你学校的网络通常有防火墙做第一层保护，外部扫描无法直接看到你实验室内部的端口。

坏消息是：如果你的电脑连接了不安全的公共 WiFi（咖啡馆、酒店、机场），你的端口就可能暴露给同一个 WiFi 下的所有人。

3. 第三层：网络安全

时长
约 6 分钟
核心
HTTP vs HTTPS、防火墙、API Key 安全

HTTP vs HTTPS：明信片 vs 密封信封

HTTP = 明信片，HTTPS = 密封信封
你寄一封信：
HTTP = 写在明信片上寄出去。任何经手的邮递员都能看到内容。
HTTPS = 装在密封信封里寄出去。只有收件人能看到内容。

当你使用 HTTP 网站时，你和服务器之间的所有数据——包括密码、API Key、研究数据——都以"明文"传输。任何在你和服务器之间的人（你的 ISP、WiFi 提供者、中间节点）理论上都能看到。

当你使用 HTTPS 网站时，数据在传输之前先加密。只有目标服务器有密钥能解密。中间人看到的是乱码。
一个简单规则，帮你避免 80% 的网络风险：
浏览器地址栏最左边有一个小锁图标 。看到它 = 这个网站用了 HTTPS。
千万不要在地址栏没有小锁的网站上输入任何密码或敏感信息。

防火墙：小区的"门禁系统"

防火墙就像小区门禁：只有持有门禁卡（被授权的流量）的人才能进入，其他人（未经授权的流量）一律挡在外面。学校、企业、甚至你个人的操作系统（Windows Defender 防火墙）都有一层防火墙。

API Key 安全 —— 全课最重要的安全提醒

API Key = 银行卡密码 —— 绝对不能泄露的四条铁律

API Key 是访问 AI 服务的"数字身份"——谁持有你的 API Key，谁就能以你的身份调用服务（产生费用，可能数万元；访问你的数据）。

铁律一：绝不写在代码里贴出去
api_key = "sk-abc123def456" 这段代码绝对不能发到 GitHub、朋友圈、微信群。

铁律二：绝不截图发到群里
很多人习惯截图问"这个报错怎么回事"——结果截图中 API Key 清晰可见。一个 500 人的微信群，瞬间 500 个人有了你的 Key。

铁律三：使用环境变量或 .env 文件 + .gitignore
把 API Key 写在 .env 文件里，把 .env 加到 .gitignore 中。代码中用 os.getenv("API_KEY") 读取。上传 GitHub 时 .env 不会跟着上去。

铁律四：不在公共 WiFi 下使用未加密的 API 调用
咖啡馆的 WiFi 密码可能是 "coffee123"，这意味着同一 WiFi 下的任何人都可能截获你的网络流量。你的 API Key 在传输中如果是明文（HTTP）就可能被窃取。

如果 API Key 已经泄露了怎么办？
立即登录服务商平台 API 管理 找到这个 Key 点击"撤销/重新生成"。Key 一旦撤销，旧 Key 立即失效。
环境变量的设置方法（Windows 示例）

方法一：系统环境变量
Win+R 输入 sysdm.cpl 高级 环境变量 新建用户变量
变量名：DEEPSEEK_API_KEY
变量值：sk-你的实际key

方法二：.env 文件（推荐）
在项目目录下创建 .env 文件：
DEEPSEEK_API_KEY=sk-你的实际key

然后在 .gitignore 中添加一行：
.env

这样你就永远不会把 API Key 上传到 GitHub 了。

公共 WiFi 的风险

风险原理防护
中间人攻击 攻击者在同一 WiFi 下伪装成路由器，截获你的所有流量 使用 HTTPS 网站、避免在公共 WiFi 上登录重要账号
恶意热点 攻击者架设一个"Starbucks_Free"热点，你连上之后所有数据被监控 确认 WiFi 名称（问店员）、使用手机热点
密码重用 你在咖啡厅登录了一个弱安全性的网站密码被截攻击者用同样密码试你的邮箱/银行/学术账号 使用密码管理器、每个网站不同密码

4. 第四层：VPN

时长
约 3 分钟
类比
VPN = 国际快递中转站

VPN 是什么？

VPN = 国际快递中转站
你在北京（私有 IP），直接发快递给纽约的客户，路上有十几个中转站，每个都能看到你的包裹信息。

你改用 VPN：先把包裹发到上海的一个中转仓（VPN 服务器），中转仓把你的包裹换个外包装、换个寄件地址（VPN 服务器的公网 IP），再从上海发往纽约。

纽约客户看到的是：来自上海中转仓的包裹。纽约客户回复 上海中转仓 重新封装 回到北京的你。

VPN 做了两件事：
1. 隐藏你真实的 IP 地址（收件人看到的是 VPN 服务器的 IP）
2. 加密传输（你的设备和 VPN 服务器之间是加密隧道）

合法 vs 非法使用

使用场景合法性说明
学校 VPN —— 在校外访问图书馆数据库 合法 你通过学校的 VPN 服务器进入校园内网，等同于你在校内上网
公司 VPN —— 远程办公访问内网 合法 公司授权员工从外部安全接入内部系统
个人 VPN —— 保护公共 WiFi 下的隐私 合法（在大多数国家） 相当于给自己的网络流量加了一层"防护膜"
用 VPN 访问被国家法律法规禁止的内容 非法 任何技术工具都不应被用来违反国家法律
重要提示：DeepSeek 等国产 AI 服务不需要 VPN

DeepSeek、通义千问、文心一言、Kimi、智谱清言等国产大模型服务均部署在国内服务器上，你直接从中国网络即可正常访问。

你不需要 VPN 就能使用这些 AI 服务。而且你的数据不经过海外网络，在数据安全上更有保障。

WorkBuddy 课程中使用的 AI 服务优先推荐国内可直连的服务。

5. 第五层：API 调用在网络层面是怎么工作的

时长
约 4 分钟
核心
API 请求的完整数据流、安全三原则
承上启下：前面四层讲了"数据怎么在网络中流动"。现在回答一个具体问题——"你用 AI 工具时，数据到底经过了哪些地方？"
一个完整的 API 调用之旅
想象你用 Claude Code 向 DeepSeek 发送一个问题："请解释 Arrhenius 方程"。数据流是这样的：

1. 你的电脑（IP: 192.168.1.5，内网地址）

2. 你的路由器/网关（IP: 公网地址，由运营商分配）——把内网地址转成公网地址（NAT）

3. 经过多个中间路由器——像快递经过多个分拣中心，每一跳检查"目的地是哪里"

4. HTTPS 加密（端口 443）——你的问题在整个路途中都是加密的，中途任何人截获都看不懂

5. DeepSeek 的 API 服务器——验证你的 API Key（"房卡对不对"） 处理请求 生成回复

6. 原路返回——同样的加密通道，把"Arrhenius 方程是 ln(k) = ln(A) - Ea/RT…"送回你的电脑

整个过程：通常不到 3 秒。经过了 10+ 个路由器。全程加密。你的 API Key 只在第 5 步被验证了一次。
科研场景中 API 安全最重要的三件事：
① API Key 绝不上传公开仓库——GitHub 有爬虫 24 小时扫描公开仓库里的 API Key，被盗刷后几小时内余额清零
② 用环境变量存储 API Key——不要把 Key 写死在代码里。用 .env 文件 + .gitignore（不上传到 GitHub）
③ 设置 API 调用限额——DeepSeek 等平台允许你设置每日消费上限，防止意外超支
API、端口、HTTPS 三者的关系：API 是 "你找谁"（DeepSeek 的服务器地址），端口是 "走哪个通道"（HTTPS 走 443），HTTPS 是 "路上有没有被偷看"（加密了，没有）。三者缺一不可。

总结：四层网络安全的"一句话"

层级概念一句话总结你记住什么？
第一层 IP & 网关 IP 是门牌号，网关是小区大门 你的数据从私有 IP 经学校网关出入互联网
第二层 端口 端口是公司分机号 443 安全，80 不安全；避免端口暴露在公共网络
第三层 安全 HTTPS 是密封信封 API Key = 银行卡密码，永不在公共环境中裸奔
第四层 VPN VPN 是国际快递中转站 合法使用（学校/公司 VPN）；国产 AI 不需要 VPN
学完这一讲的你应该具备的网络安全常识：

知道自己的数据流经了哪些环节（你的电脑 网关 互联网 目标服务器）
能识别安全的网络连接（URL 前有 小锁 = HTTPS = 安全）
API Key 绝不在代码/截图/群聊/公共 WiFi 中暴露
知道 VPN 的正确和错误用法
理解 localhost:3000 中的 3000 是"端口的第 3000 号分机"

这些不是"程序员才需要知道的"。这些是每个用 AI 工具做科研的人都应该知道的——因为你的研究数据、你的 API Key、你的学术成果，都跑在这些网络上。
