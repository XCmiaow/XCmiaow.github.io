1. Skills 深度剖析

"Skill = 手机 App"
你手机上有各种 App——每个 App 专注于完成一类任务。WorkBuddy 的 Skill 就是"AI 版的 App"：每个 Skill 封装了一个完整的专业工作流，一个指令就能触发。你不会用"记事本"去修图，同样，你不会用一个通用 AI 去处理催化论文——你会用"催化论文润色 Skill"。

Skills 的三个来源

📦
内置 Skills（20+）
WorkBuddy 自带的基础 Skills：代码审查、文件管理、文档处理等。无需安装，即开即用。
🏪
SkillHub（70K+）
社区共享的 Skills 市场。搜索并一键安装。覆盖科研、工程、金融、教育等各领域。
自定义 Skill
根据你的专业需求自己创建的 Skill。完全掌握专业逻辑和输出规范——这是最核心的能力。

Skill 的文件结构

文件/目录作用示例（催化论文润色 Skill）
SKILL.md核心：Skill 的触发词、工作流、输出规范触发词："润色/润色催化/polish catalytic" 执行 ACS 风格润色流程
scripts/配套脚本（Python/Shell 等）术语检查脚本：检测"catalyst activity"是否应改为"catalytic activity"
references/参考资料库ACS 催化论文风格指南 PDF、常用术语对照表、期刊缩写列表
assets/模板/素材文件Figure 标注模板、Scheme 标准格式

创建一个专业 Skill：四个要素

演示：创建"催化论文润色 Skill"

① 名称："催化论文润色-ACS风格"
② 触发词：催化润色、polish catalysis、ACS润色
③ 工作流：
  步骤1：检查术语一致性（如"catalyst" vs "catalytic"、"conversion" vs "yield"）
  步骤2：按 ACS Catalysis 风格指南润色每段
  步骤3：检查图表标注格式（Figure 1. 大写 F，Scheme 1. 大写 S）
  步骤4：输出修改对照表（修改前 vs 修改后 + 修改原因）
④ 输出规范：Markdown 格式，左侧原文，右侧修改版，底部列出术语替换表
创建 Skill 的核心原则：
① 明确单一职责——一个 Skill 只解决一类问题
② 工作流可重复——每步逻辑清晰，不论输入如何都能按同样流程处理
③ 术语专业——引用你所在领域的标准术语和规范
④ 允许迭代——第一版不求完美，用起来再改进

2. MCP 协议实战

"MCP = 万能转接头"
你出国旅行时，各种插头规格不同（英标、欧标、美标、国标），一个万能转接头解决所有问题。
MCP（Model Context Protocol）就是 AI 的万能转接头——它让 WorkBuddy 能连接任何遵循 MCP 标准的第三方工具和服务。

MCP 的三个角色

🖥️
MCP Server
第三方工具对外暴露的接口服务。相当于"这个工具对外开放的 API 接入点"。
MCP Client
WorkBuddy 本身。它通过 MCP 协议发现和调用各种 MCP Server 的能力。
👤
你
你不需要理解 MCP 的技术细节。你只需要告诉 WorkBuddy"帮我连上 X 工具"，AI 自己配置。

核心 MCP 功能卡

Playwright
浏览器自动化。自动打开网页、截图、填写表单、抓取数据。用于：自动获取期刊最新目录、下载论文摘要、抓取化合物数据库。
🗄️
Database
数据库连接。直接查询 SQLite/MySQL/PostgreSQL。用于：查询实验数据库、关联多表数据、批量更新。
🐙
GitHub
代码协作。创建 Issue、提交 PR、查看 CI 状态。用于：管理课题组代码库、协作开发。
📁
File System
文件系统操作。读写、搜索、批量重命名。用于：数据文件管理、日志分析、批量转换格式。
优先级建议：File System + Playwright = 覆盖 80% 的科研需求。先掌握这两个，再逐步扩展到其他 MCP。

配置 MCP 的两种方式

方式适合人群操作
可视化界面初学者WorkBuddy 设置 MCP 管理 点击添加 从列表选择 一键配置
手动 JSON进阶用户编辑 mcp.json 配置文件 填写 Server 地址、参数 重启 WorkBuddy
实操任务：配置浏览器 MCP + 抓取最新期刊目录
① 在 WorkBuddy 中配置 Playwright MCP Server
② 输入指令："打开 ACS Catalysis 最新一期的目录页，抓取每篇文章的标题、作者、DOI，保存为 CSV"
③ 观察 AI 自动打开浏览器 导航 提取数据 保存文件的完整过程

3. 多 Agent 协作

"一个团队同时干活"
不是"一个 AI 帮你做一件事"，而是"同时启动 4 个 AI Agent，每个专注一个子任务，最后汇总结果"。
就像你让 4 个博士生同时查找 4 个不同方向的文献，你去开会回来他们就把整理好的材料放在你桌上了。

真实案例：并行文献调研

场景：你要写一篇关于"CO₂ 加氢制甲醇催化剂"的综述，需要覆盖 4 个维度。

Agent A（催化机理）："搜索并总结近 5 年关于 Cu/ZnO 催化剂在 CO₂ 加氢中的活性位点和反应机理的研究，列出 5 篇关键论文"

Agent B（新型材料）："搜索和总结近 5 年用于 CO₂ 加氢制甲醇的新型催化材料（In₂O₃、金属有机框架、单原子等），列出 5 篇关键论文"

Agent C（交叉学科）："搜索将机器学习或 DFT 高通量筛选应用于 CO₂ 加氢催化剂设计的研究，列出 3 篇代表性论文"

Agent D（工业案例）："搜索 CO₂ 加氢制甲醇的工业示范项目和技术经济分析，列出 3 个关键案例"

结果：15 分钟后，4 个 Agent 同时返回结果——一份覆盖机理、材料、计算、工业的完整文献调研。

什么时候用多 Agent？

适用条件：
① 任务可以自然地拆分为 2-4 个独立子任务
② 每个子任务有 明确边界，不会互相依赖
③ 每个子任务 "单独也能完成"，不需要中途沟通

反例（不适合多 Agent）："先清洗数据，然后根据清洗结果做统计，再根据统计结果画图"——这是一个依赖链条，不适合并行。

Agent 架构深度剖析

Agent 之间怎么协作？—— 一张图讲清楚
想象一个项目组开会：

你（Leader） 分配任务 Task List（共享任务看板）
                            ↙            ↘
                Agent A  Agent B  Agent C
                 ↕ 消息传递 ↕
                Message Bus（消息总线）

三个关键机制：
① Task List 共享——所有 Agent 看到同一张任务表。谁认领了哪个任务、谁做完了、谁被卡住了——一清二楚。
② Agent 间消息传递——Agent A 可以给 Agent B 发消息："你那个数据查完了吗？我这边等你结果。"——就像项目组里同事互相催进度。
③ Leader-Worker 模式——你永远是 Leader，Agent 是 Worker。它们可以互相通信和协调，但最终决策权在你手里。
串行 vs 并行：什么时候用多个 Agent
• 串行（一个Agent做完另一个才做）：A 的输出是 B 的输入——比如"A 先搜文献 B 再基于结果写综述"
• 并行（多个Agent同时做）：多个独立任务——比如"A 搜催化文献""B 搜材料文献""C 搜方法论文献"同时进行

判断法则：如果子任务之间没有依赖关系（互不等待），就并行。如果有依赖，就串行。不要为了"看起来酷"而强行并行——一个停摆的 Agent 会卡住整个流程。

Cron 定时任务：让你的 AI 在你不在的时候干活

⏰ 什么是 Cron
Cron 是 Unix/Linux 系统中的定时任务调度器。简单说："设好闹钟，时间到了自动执行"。

类比：实验室的自动采样器。你不需要通宵守在仪器旁边每 30 分钟手动采集一次数据——设好程序，仪器自动采样。Cron 就是 AI 世界的"自动采样器"。

WorkBuddy 中的 Cron：你可以设置"每周一早上 9:00 自动总结上周实验数据""每月 1 号生成上月文献阅读报告""每天晚上 6:00 检查是否有新发表的论文邮件"。设好一次，永远不用管。

随堂练习：设计一个你的 Cron 任务

① 这个任务是什么？
② 触发频率（每天/每周/每月）？
③ WorkBuddy 需要自动做什么？
④ 结果发送到哪里（文件/群消息/邮件）？

4. 远程遥控 + 自动化

远程遥控

场景：你正在外地开会，手机收到消息——课题组需要 XRD 数据分析。

操作：
① 绑定企业微信/微信机器人到 WorkBuddy
② 从手机发送指令："分析昨天新测的 XRD 数据，识别物相，计算结晶度，画比对图"
③ 你的 PC 上的 WorkBuddy 接收指令 读取文件 分析 生成图表 把结果回传到你的手机

不需要远程桌面，不需要带电脑，一条消息搞定。

Channel 通道机制：你的手机怎么指挥电脑

📡 什么是 Channel
Channel（通道）是 WorkBuddy 接收远程指令的"通信管道"。

类比：快递公司。你不在乎快递是顺丰还是圆通——你在乎的是"东西能送到"。企业微信、QQ、飞书、钉钉——这些都是不同的 Channel（快递公司），WorkBuddy 后端统一接收（收件箱）。

四种 Channel 对比：
Channel配置难度延迟推荐场景
企业微信 最快 1 分钟最低首选——高校教师通常已有企业微信
QQ中等个人用户、学生群体
飞书中等使用飞书的企业/团队
钉钉中等使用钉钉的学校/团队
Channel 的工作原理（一句话版）：
你的手机 企业微信 App 把消息发给 WorkBuddy 后端服务器 服务器验证你是谁 找到你登录的电脑 把指令发给电脑上的 WorkBuddy WorkBuddy 执行 结果回传 你手机上收到回复。

前提条件：电脑必须开机 + WorkBuddy 保持运行 + 网络连接正常。电脑关机了，手机喊破嗓子也没用。

自动化

定时任务示例：
- "每周一上午 9 点：汇总上周实验数据，生成周报草稿"
- "每天下午 5 点：检查 ACS Catalysis 是否有新论文上线，有的话发摘要到微信群"
- "每月 1 号：整理上月实验室化学品库存，标记低于阈值的试剂"

设置方式：在 WorkBuddy 的自动化页面设置定时规则（类似手机闹钟），绑定执行的指令。

5. 讨论

讨论主题：每个人列出"我的科研工作中最可以自动化的 3 个任务"。

思考框架：
- 这个任务是否重复发生？（每周/每月都做 可自动化）
- 这个任务是否规则明确？（步骤固定，不需要每次重新判断 可自动化）
- 这个任务是否占用大量时间但价值不高？（超过 2 小时/周 值得自动化）

导师引导：把每个人的 3 个任务分类，形成"实验室自动化优先级清单"，作为结课作业的参考。

本讲小结

  • Skills 三层来源：内置（20+） SkillHub（70K+） 自定义。核心能力是创建自己的 Skill
  • Skill 文件结构：SKILL.md（核心流程）+ scripts/（脚本）+ references/（知识库）+ assets/（模板）
  • MCP = AI 的万能转接头：优先掌握 File System + Playwright，覆盖 80% 科研需求
  • 多 Agent 协作：4 个 AI 同时处理 4 个独立子任务，适用于横向并行（非纵向依赖链）
  • 远程遥控 + 自动化：手机发指令控制 PC，定时任务自动执行重复性工作
